# 数据中台 — 业务逻辑设计

本文档说明「**数据中台**」路由下 **`DataSync`** 页的能力：外部**数据源/接口**配置、**cURL 导入**、**联调测试**、**响应体**在 **业务行 / 原始 JSON / 清洗后的 JSON** 三视图下浏览与**字段展示配置**，以及**单元格详情抽屉**。清洗后的 JSON 支持 **大模型（自然语言）** 与 **脚本（本地分组）** 两种方式。主代码：`src/components/DataSync.tsx`；状态与行为见 `externalApiStorage`、`dataSyncResponseStorage`、`dataHubJsonCleaning`、`dataHubScriptCleaning` 等。

---

## 1. 功能定位

- **与「销售预测/报告管理」中的 DataHub 关系**：**同一套** 接口配置 与**最近一次响应** 可被 `loadDataHubState` / `loadDataSyncLastBody` 等在**报告、计划、助手**中**读取**，实现「**一次配置、多处复用**」。  
- **CORS**：浏览器直连接口可能受 CORS 限制，文档注释中说明需 **Vite 代理**或**后端转发**。  

---

## 2. 信息架构

- **按数据源（平台）分组**：`DataPlatform` 下挂多个 `ExternalApiProfile`（`types/externalApiProfile.ts`），每一配置含 URL、方法、头、**清洗方式**（`jsonCleaningMode`）、**自然语言清洗规则**（`jsonCleaningRules`）、**脚本分组规格 JSON**（`jsonCleaningScriptSpec`）等。  
- **主 Tab**  
  - **基础/接口**（`basic`）：**cURL 粘贴解析**（`parseCurl`）→ 快速填充配置；**测试请求**、耗时与 HTTP 状态、错误、**截断长响应**预览。  
  - **数据**（`data`）：在选定接口上展示**三类子 Tab**。  

---

## 3. 数据三子 Tab

| 子 Tab | 含义 |
|--------|------|
| **业务数据** | 将 **原始 JSON** 用 **启发式/固定路径** 抽为**行表** `extractBusinessRowsFromJson`；支持列筛选、搜索、**可见列**与**拖拽列序**（`buildDataHubFieldPickerOrder`）。 |
| **原始 JSON** | 上次测试返回的**原文**（或从存储恢复），便于对照。 |
| **清洗后的 JSON** | 见下文 **§3.1**；结果均为合规格式化 JSON 文本；**会话级**写入 `session` 前缀 **DATA_HUB_CLEANED_JSON_***（`dataHubCleanedJsonStorage`），供报告/解析链路 **按 profileId 读取**。 |

### 3.1 清洗后的 JSON：两种模式

| 模式 | 字段 | 行为 |
|------|------|------|
| **脚本（默认主路径）** | `jsonCleaningMode === "script"`；`jsonCleaningScriptSpec` 为 JSON 字符串 | **不调用大模型**。由 `dataHubScriptCleaning` 解析规格，对**原始 JSON** 做确定性变换；适合长响应、避免模型输出被截断。当前内置一种脚本：`kind: "groupBranchWorkshopDate"`（`version: 1`）。**新建接口**默认 `script` 并带默认 `jsonCleaningScriptSpec`。 |
| **大模型** | `jsonCleaningMode === "llm"`；`jsonCleaningRules` 为自然语言 | 依赖 `readLlmEnv`；`buildJsonCleaningUserMessage` + `callLlmChatText`（系统提示见 `dataHubJsonCleaning`）；`formatCleanedJsonOutput` 整理输出。长 JSON 仍受 **max_tokens / 输出截断** 影响（`DATA_HUB_JSON_CLEANING_MAX_COMPLETION_TOKENS` 等）。 |

**脚本分组（`groupBranchWorkshopDate`）约定**

- **输入假设**：在「列表路径」下得到**数组**；每一项再经「行对象路径」得到**扁平行对象**（如 `data.list[].variables`），行上含用户配置的四个**字段名**：分公司、车间、日报日期、正文/详情。  
- **列表路径**：点分键名，从根对象走到数组（如 `data.list`）；**留空**表示根节点自身即为数组。  
- **行对象路径**：点分键名，从列表元素走到行对象（如 `variables`）；**留空**表示列表元素本身即为行对象。  
- **四个分组/正文字段名**：须与**原始 JSON 中行对象**上的**键名**一致，可在界面中改为接口实际字段名（如 `分公司名称` 与 `所属分公司` 二选一取决于源数据）。  
- **输出形状**：顶层数组，元素为 `{ "分公司名称", "日报日期", "车间日报列表": [{ "车间名称", "日报详情" }] }` —— 先按 **(分公司, 日期)** 聚合，同组内多车间、同车间多段正文分别合并。  
- **默认表单建议值**（典型中台 list + variables）：`listPath=data.list`，`itemPath=variables`，字段名 `所属分公司` / `所属车间` / `日报日期` / `日报内容`。见 `DEFAULT_GROUP_SCRIPT_SPEC`。  
- **迁移**：持久化中未写 `jsonCleaningMode` 时，若已有非空 `jsonCleaningRules` 则视为 **大模型**；否则视为 **脚本**并在缺省脚本规格时注入 `DEFAULT_GROUP_SCRIPT_SPEC`（见 `externalApiStorage.normalizeProfile`）。  
- **边界**：若数据已是「按分公司+日期嵌套车间列表」的二次结构，本脚本**不**负责展平再聚合；需另增脚本类型或先用大模型打成扁平行再跑脚本（产品决策）。

**保存与触发**

- 页内 **「保存清洗规则」** 写入：`jsonCleaningRules`、`jsonCleaningMode`、`jsonCleaningScriptSpec`（脚本规格为 `JSON.stringify` 表单对象）。  
- **测试成功（2xx）** 时同步写入上述三项**当前草稿**，并触发与保存后相同的重算逻辑（见 `DataSync` 中 `handleRunTest` / 清洗 `useEffect`）。

## 4. 与 LLM/配置

- **大模型环境**与**设置**中 API Key/模型**同源**（`readLlmEnv`、`LLM_CONFIG_CHANGED_EVENT`），修改后数据中台**无需**另存 key；**仅在选择「大模型」清洗模式时**需要有效环境。  
- **脚本模式**不读 LLM 配置。  
- 大模型清洗有 **max tokens** 等限制常量（`DATA_HUB_JSON_CLEANING_MAX_COMPLETION_TOKENS` 等）避免超长**模型输出**被截断；脚本模式无此限制（受浏览器解析/序列化体积约束）。

## 5. 持久化

| 内容 | 存储 |
|------|------|
| 平台+接口配置列表（含 `jsonCleaningMode`、`jsonCleaningScriptSpec`） | `externalApiStorage`（`saveDataHubState` / `loadDataHubState`） |
| 最近测试响应体等 | `dataSyncResponseStorage`（如 `saveDataSyncLastBody`） |
| 清洗结果（会话，键前缀） | `dataHubCleanedJsonStorage` |
