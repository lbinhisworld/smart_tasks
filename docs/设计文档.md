# 设计文档（数据看板、任务模型与规则）

本文档描述首页「数据看板」中与**当前视角**相关的数据范围与 UI 文案规则，并汇总**任务编码**、**任务分类**、**任务状态**等与产品一致的设计约定，便于产品与实现对照。实现位置见各节引用路径。

## 1. 首页数据看板结构

- **任务看板**：任务列表、范围筛选、风险与分类统计等；任务可见性由 `taskVisibleForPerspective` 决定（见 `src/utils/leaderPerspective.ts`）。
- **报告看板**：`ReportDashboardTab`，展示按提取日聚合的产量 KPI 树（集团 / 分公司 / 车间）及原文引用侧栏。父组件 `Dashboard` 在「报告看板」Tab 下从本地存储加载提取历史，并按视角过滤后传入子组件。

相关入口：`src/components/Dashboard.tsx`。

## 2. 当前视角（领导视角）

- 顶部可选视角包括固定项 **「集团领导」** 以及部门架构中每一行对应的 **「{架构单元}领导」**（由 `buildPerspectiveOptions` 生成）。
- **集团领导**：组织单元为「全集团」，不对报告提取历史做分公司维度过滤。
- **分公司领导**（架构名以「分公司」结尾，如「广西分公司领导」）：报告提取历史仅保留 `pickBranchCompany(item)` 与当前分公司一致的数据。
- **非分公司职能部门领导**：报告提取历史仍看**全部**记录（与集团领导在报告维度上的数据全集一致），与 `extractionHistoryVisibleForPerspective` 行为一致。

过滤函数：`extractionHistoryVisibleForPerspective`（`src/utils/leaderPerspective.ts`）。  
`Dashboard` 使用：`filteredReportHistory = reportHistory.filter(h => extractionHistoryVisibleForPerspective(h, user.perspective))`。

## 3. 报告看板：一级「当日产量指标」括号内说明

一级汇总卡片标题格式为：

> **当日产量指标（{scopeLabel}）**

`scopeLabel` 须与**当前视角下实际参与聚合的提取历史范围**一致（即与传入 `ReportDashboardTab` 的 `history` 一致），规则如下：

| 当前视角 | `scopeLabel`（括号内） |
|----------|------------------------|
| 集团领导 | 全部分公司汇总 |
| 非分公司职能部门领导（可看全量报告历史） | 全部分公司汇总 |
| 分公司领导（如「广西分公司领导」） | **{分公司名}汇总**（如「广西分公司汇总」） |

实现：`reportDashboardLevel1ScopeLabel(perspective)`（`src/utils/leaderPerspective.ts`）。  
展示：`ReportDashboardTab` 接收 `perspective`，标题为 `当日产量指标（${level1ScopeLabel}）`；`Dashboard` 传入 `perspective={user.perspective}`（`src/components/ReportDashboardTab.tsx`、`src/components/Dashboard.tsx`）。

## 4. 与其它文案的关系

- 原文引用侧栏等场景若仍使用「全集团汇总」等固定文案，属于**引用展示维度**的命名，可与上述一级卡片括号说明独立演进；若产品要求全文案统一，需单独评审后改 `ReportCitationDrawer` / `reportCitation` 等相关逻辑。

## 5. 任务编码规则

自动编号在 **`addTask`** 中生成（`src/context/TaskContext.tsx`），逻辑见 **`src/utils/taskCode.ts`**。手工传入 `code` 时可覆盖。

### 5.1 格式

```
QF-{发起部门2位}-{大类3位}-{流水4位}
```

示例：`QF-XZ-HSE-0001`（发起部门「行政部」、大类「安全环保管控类 (HSE)」、流水 1）。

### 5.2 发起部门 2 位（拼音首字母，大写）

- 对部门名称去首尾空白后，若仅含字母则取**前两个英文字母**；仅一个字母时第二位补 `X`。
- 否则先去掉常见组织后缀（如：`部`、`分公司`、`有限公司`、`公司`、`集团`、`领导`、`车间`、`处`、`室`、`厂`、`中心`、`小组`），再在剩余字符串中取**前两个汉字**，用 **`pinyin-pro`** 取拼音**首字母**并大写，拼接为 2 位。
- 无法得到有效结果时用 `NA`。

### 5.3 大类 3 位（字母）

与 `categoryLevel1` 精确匹配，映射表见 **`TASK_CATEGORY_LEVEL1_CODE3`**（`src/data/taskCategories.ts`）：

| 任务大类（level_1，与界面一致） | 三位码 |
|--------------------------------|--------|
| 安全环保管控类 (HSE) | HSE |
| 生产能效对标类 (Production) | PRO |
| 质量专项攻坚类 (Quality) | QUA |
| 设备本质安全类 (Maintenance) | MNT |
| 管理作风与赋能类 (Management) | MAN |

未在表中的新大类：尝试从名称括号内英文取前 3 位（不足补 `X`）；仍无法解析则为 **`GEN`**（`taskCategoryLevel1Code3`）。

### 5.4 四位流水号

- 在**当前任务列表**中，对所有 **`QF-XX-YYY-NNNN`**（2 位 + 3 位 + 4 位数字）格式的 `code` 解析末 4 位，取最大值 **+1**。
- **旧版编号**不符合该正则的不参与统计，避免与历史数据混排时错位；若列表中尚无非旧格式编号，则从 `0001` 起。

## 6. 任务分类规则（大类 / 子类）

数据权威来源：**`TASK_CATEGORY_RULES`**（`src/data/taskCategories.ts`）。

- **大类**：与每条 `level_1` 字符串**完全一致**（含括号内英文）。
- **子类**：必须为对应大类下 `level_2` 中某一元素的 **`name`**，与界面、存库、`generateTaskFromDailyPlanRow` 中 LLM 约定一致。
- 每个子类含 **`extraction_guide`**（关注场景与关键词），用于日报计划生成任务时随系统提示**整包 JSON** 注入大模型，便于 `categoryLevel1` / `categoryLevel2` 精准归类。完整 JSON 由 `formatTaskCategoryRulesForLlm()` 序列化。

### 6.1 大类与子类一览（名称）

**安全环保管控类 (HSE)**  
子类：隐患整改项；违章行为纠偏；季节性防御；合规性核查。

**生产能效对标类 (Production)**  
子类：产量达标跟踪；能效单耗压降；损纸消纳专项；成本控制分析。

**质量专项攻坚类 (Quality)**  
子类：纸病消除专项；工艺指标优化；客户反馈闭环；新品/实验跟踪。

**设备本质安全类 (Maintenance)**  
子类：预防性维护；技术改造攻坚；备件及资产管理；外部干扰防护。

**管理作风与赋能类 (Management)**  
子类：标准执行闭环；技能培训考核；精益标准化。

各子类 **`extraction_guide`** 全文以源码为准（同上文件），本文不重复粘贴，避免与实现漂移。

### 6.2 与任务编号的关系

任务编号中的「大类 3 位」仅依赖 **`categoryLevel1`**，与子类无关；子类仍须在上述集合内选择。

## 7. 任务状态类型与展示

类型定义：**`TaskStatus`**（`src/types/task.ts`）。下拉/编辑中的顺序见 **`STATUSES`**（`src/context/TaskContext.tsx`）：进行中 → 实质性进展 → 卡住待协调 → 已完成。

| 状态 | 列表与详情中的标签色（白字） |
|------|------------------------------|
| 已完成 | 绿 `#0d9f6e` |
| 实质性进展 | 蓝 `#1d6bc6` |
| 进行中 | 橙 `#ea580c` |
| 卡住待协调 | 红 `#dc2626` |

- **任务列表**中状态为**只读彩色标签**，改状态在「编辑任务」或「手工新建任务」中操作。
- **任务详情**侧栏顶栏标题为**状态文字**，背景色与上表一致（`taskDetailDrawerToolbarModifierClass`，`src/components/TaskStatusPill.tsx` / `App.css`）。

### 7.1 「卡住待协调」与协调方

- 当 `status === "卡住待协调"` 时须设置 **`coordinationParty`**，取值限定为：**集团公司**、**各职能部门**、**分公司领导**（`COORDINATION_PARTY_OPTIONS`）。
- 存库与归一化逻辑见 **`normalizeStoredTask`**（`TaskContext`）：非该状态时清除协调方。

### 7.2 领导指示（`leaderInstruction`）

- 可选字段；**日报计划生成任务**时，将合并后的领导指示文本写入（`generateTaskFromDailyPlanRow`）。
- **报告 → 现有任务进度更新**中，模型可在 JSON 中返回 `leader_instruction_line`，非「暂无」时回写任务（`llmInferTaskProgressFromDaily.ts`、`ReportManagement.tsx`）。
- 详情页以**红底白字圆角卡片**展示（`task-detail-leader-card`）；任务动因 / 任务描述为蓝底白字卡片（`task-detail-highlight-card`）。
