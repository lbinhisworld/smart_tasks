# 销售预测 — 业务逻辑设计

本文档描述本项目中「销售预测」功能的业务目标、数据流、计算规则与界面行为。实现代码主要位于 `src/components/SalesForecast.tsx` 及 `src/utils/` 下若干模块。

---

## 1. 功能定位

- **用户显示模式 / 调试显示模式**（双 Tab，偏好单独持久化）：对外可保留精简的「用户显示模式」；**数据准备、CSV 上传、底表、数量分类、客户进货周期性分析、各类计算**均在「调试显示模式」中完成（当前以调试 Tab 为主战场）。
- **本地优先**：核心状态写入浏览器 **localStorage**，刷新或切换应用内页签后，在未清除站点数据的前提下可恢复（受配额与浏览器策略约束）。

---

## 2. 数据流总览

从上游到下游的典型链路如下：

1. **上传并保存 CSV** → 在内存中形成 **销售数据预览**（表头 + 行矩阵，首行为表头）。
2. **拆解物料记录**（按钮）→ 从预览构造 **销售分析底表**（行类型与物料解析、字段映射一致）。
3. 可选：在底表上 **生成数量分类**（对「数量」列分档，写回一列并展示分档图）。
4. **启动进货周期性分析**（按钮）→ 从底表行聚合为 **客户进货周期性分析**用多级树（客户 + 型号 → 品名 → 规格 → 克重，五级指标位；**见第 6 节：只保留可解析维度**）。
5. 在该分析中按需点击：**计算下单次数**、**计算订货间隔标准差**、**计算订货间隔平均值**、**生成周期性标签**（各按钮写入或覆盖对应节点上的字符串字段，并随持久化包保存；其中「下单次数 / 订货间隔标准差」含**多轮分步计算**与**完成态**持久化，见第 7、8、9 节）。

---

## 3. 销售数据预览

- **文件**：`decodeTextBytesAuto` 解码（支持 UTF-8 与常见 GBK/GB18030），`parseCsvText` 解析，支持引号内逗号。
- **表头**：首行作为表头；空表头会生成占位列名。
- **列宽**：对上传矩阵按表头列数做补齐/截断，与表头列数一致。
- **客户列展示**：若表头中能找到与「往来户名称」等价的列，保存预览时会对该列做**脱敏/格式化**展示（`formatCustomerPreviewName`），不改动源文件内容逻辑以外的业务行（仅影响写入内存的预览行）。
- **持久化**：成功解析后与后续分析状态一并写入 `salesForecastStorage` 中的 preview 段。

## 4. 销售分析底表

**模块**：`src/utils/salesAnalysisBaseFromPreview.ts`

- **行模型 `SalesAnalysisBaseRow`** 包含：日期、客户名称、销售组、业务员、**物料标签数组**、物料合并编码、数量等（与表头**约定字段名**映射）。
- **列映射**（表头中按候选名**模糊/精确**查找）：如「单据日期」→ 日期、「往来户名称」→ 客户名称、「物料合并」→ 合并编码、「数量」→ 数量等；缺列时对应字段可能为空，并给出 **missingHint** 提示未匹配到的源字段。
- **物料标签**：由物料合并编码、物料描述等经 `parseMaterialCode` 等解析，得到带 `kind`（如 model/name/spec/grammage 等）与 `text` 的 **MaterialTag** 列表，用于**分析树**分级下钻的键**与**底表行过滤**（见 `dimModelKey` / `dimNameKey` / `dimSpecKey` / `dimGrammageKey`）。
- **客户列索引**：`findSalesCustomerSourceColumnIndex` 用于预览阶段对客户列的格式化，与上表头约定一致。

底表是后续 **数量分类** 与 **客户进货周期性分析、订货间隔、下单次数** 的**共同数据源**（除数量分类仅强依赖数量外，**分析树**与间隔均强依赖底表行与标签）。

## 5. 数量分类

**模块**：`src/utils/calculateOrderSegments.ts`

- 输入为底表每行的**数量**（解析为正数后参与计算）。
- **分档思想**：在有效订单上，按**单笔数量升序**累计进货量，达到**总量 10%** 时的那笔作为「**零散**」上界；按**降序**累计达到**总量 70%** 时的那笔作为「**高**」下界，中间为「**低**」。有效样本不足 10 条时，使用基于均值的**备用阈值**。
- **打标规则**（与 UI 文案一致）：零散为严格小于零散界；低为两界**闭区间**；高为严格大于高价值界。无法解析或非正数则不进入三档。
- **与界面**：在调试 Tab 中生成后，在底表旁展示阈值说明与分档小图，并在表内增加一列（若已生成则插入「进货量分类」列）。结果参与 **localStorage** 全量包保存。

## 6. 客户进货周期性分析

**模块**：`src/utils/buildCustomerInboundDimensionFromAnalysis.ts`

### 6.1 树形结构

- **第一层：客户**（去重、排序），每个客户下为 **型号 → 品名 → 规格 → 克重** 四级，与物料标签的 model / name / spec / grammage 对应。
- **只展示有底表行支撑的路径**；且 **仅当该层维键为「可解析」时才生成节点**（见下节）。仍仅包含有订货记录的组合在底表中的数据；剪枝后若某客户下无型号则**不保留该客户块**。

### 6.2 可解析的型号、品名、规格、克重

- **分析树**分组键为 **`dim*Key` 的返回值**；与 **`DIM_MISSING_LABEL`（`"-"`）** 相等的层视为**该维未从标签解析到有效值**（或仅占位），**不创建该层节点**。
- **克重**列表只包含键 **≠** `"-"` 的组；若某**规格**下克重子项全部被过滤，则**不保留该规格**；**品名 / 型号 / 客户**同理向上剪枝，直至某层有至少一个可解析子树。
- 本功能界面上**不会出现**「未解析」规格/克重占位卡片；与「可解析的物料维度」展示一致。旧版 `localStorage` 中可能仍含历史占位，**重新点击「启动进货周期性分析」** 可按新规则重建。

### 6.3 每节点指标位（`CustomerInboundDimensionMetrics`）

| 字段 | 含义（业务上） |
|------|----------------|
| 最近一次下单日期 | 该节点汇总范围内，底表「日期」解析后的**最大**日期，格式化为 YYYY-MM-DD 等可展示形式；无有效日期时可为占位 |
| 下单次数 | 见第 7 节**两轮分步**；初始生成时可用底表行数等占位，由「计算下单次数」按规则覆盖 |
| 订货间隔标准差、平均值 | 见第 8 节 |
| 周期性标签 | 见第 8 节 |

### 6.4 树操作与展示

- **客户折叠**：只隐藏**子树**（型号及以降）；**本客户行的五列指标**始终显示。
- **型号 / 品名 / 规格折叠**：只隐藏**下级**列表，**本行五列指标**始终显示。
- **克重**为最末级，无下钻，左侧为占位对齐有折叠的层级。
- 公司级/各级卡片有约定的色彩与**标签图标**（`CustomerDimLabelIcon`），与物料标签色体系一致。

### 6.5 与「启动进货周期性分析」

- 从当前底表行 `buildCustomerInboundDimensionFromAnalysis(rows)` 一次生成**整棵分析树**与初始指标占位；生成后**默认展开所有客户**（`openCustomerDimIds` 行为以 `SalesForecast` 为准）。  
- 重新启动会替换内存中的 `customerInboundDimension` 并持久化，并**将「下单次数 / 订货间隔标准差」完成态标志置为未达成**（见第 9 节）。

## 7. 下单次数

**模块**：`src/utils/applyOrderCountToDimension.ts`；交互在 `SalesForecast` 的「计算下单次数」按钮。

**两阶段、多步写回**（总步数 = 克重步数 + 规格 + 品名 + 型号 + 客户，进度为**单序列**）：

1. **第一轮：仅最底层「克重」**  
   对**销售分析底表**在「客户 + 型号 + 品名 + 规格 + 克重」路径下**过滤行**，该路径下**行数**即该克重卡片上的**下单次数**（字符串化）。克重**路径顺序**与 `listGrammagePathsInTreeOrder` 一致（**分析树自上而下、深度优先**，与 `CustomerDimensionModelTree` 渲染序一致）。实现上为 `listGrammageOrderCountStepsFromBase` 复用该路径后逐格计数。

2. **第二轮起：自下而上加总**（克重已写入后，按树序写**规格** → **品名** → **型号** → **客户**）  
   每一父节点 = **直接子节点**的「下单次数」**可解析为整数的**求和（空、`—`、非数忽略，见 `orderCountByAggregatingFromChildren` + `setOrderCountOnDimension`）。

- **分步与刷新**：`requestAnimationFrame` 分步，每步更新 `setCustomerInboundDimension` 以刷新格内数字；**持久化在整段结束后**写一次。  
- **与「计算订货间隔标准差」**等**互斥**：进行中时相关按钮/操作需等待或逻辑内 return，避免同时跑多轮计算（见 `SalesForecast`）。  
- **完成态与刷新**：`saveSalesForecastPersisted` 的选项 **`orderCountComputeCompleted: true`** 表示已成功完成本轮并落盘；**刷新或切页回来** 从 `loadSalesForecastPersisted` 读回该标志，**绿底白字**按钮态与之绑定；**新 CSV、拆解底表、重新启动进货周期性分析** 等会写入 **`false`** 并清状态。仅「计算进行中」为忙态，**不**因另一路统计的忙而灰掉本按钮的样式时，以源码为准。  
- **保存失败**：若末次保存失败，完成态不写入持久化，界面不维持成功态（与 `SalesForecast` 实现一致）。

## 8. 订货间隔与周期性

**核心模块**：`src/utils/customerOrderIntervalStats.ts`；`applyOrderCountToDimension.ts` 中 **`setOrderIntervalStdDevOnDimension`** 按路径写入标准差格。

**时间、间隔、样本统计**（**克重**及**从底表直接回退**的层）：取底表行「日期」列解析为日序，相邻两笔的**自然日**差，再求样本标准差/均值等（`computeOrderIntervalStatsFromRows`、`formatIntervalDaysStat`，与 `localMidnightMs` 等实现一致）。客户键与**底表-分析树**过滤**同一套** `filterRowsForCustomer` 等归一化。

### 8.1 界面主路径：「计算订货间隔标准差」

分**多轮**与「下单次数」类似（总步 = 全克重 + 全规格 + 全品名 + 全型号 + 全客户）：

1. **第一轮：克重**  
   对每个克重路径**当场**用底表行过滤后算样本标准差（`computeOrderIntervalStdDevForGrammagePath`），写回后 `setCustomerInboundDimension`，顺序同 **`listGrammagePathsInTreeOrder`（**分析树**树序，非先整表预扫）**。

2. **后续：规格 → 品名 → 型号 → 客户**  
   - **规格**、**品名**格：不一律用**子级标准差展示串的算术平均**；而使用 **带「底表回退」的汇聚**（`orderIntervalStdDevForSpecPathWithFallback` / `orderIntervalStdDevForNamePathWithFallback`）：
     - 若子级存在 **未解析占位**（如历史数据中有 `"-"`）或 **任一直接子**的「订货间隔标准差」**无法解析为有效数字**（空、`—`、`-` 等）→ 对该**规格/品名**格改为对**销售分析底表**按**该级路径**（客户+型号+品名+规格，或客户+型号+品名）**全量筛行**后，用**同一套** `computeOrderIntervalStatsFromRows` 算样本标准差。  
     - 否则：仍为**可解析子格**的**算术平均**（`averageStdDevDisplayStrings`），两位小数。  
   - **型号**、**客户**：仍用**直接子**标准差串的**算术平均**（`averageOrderIntervalStdDevFromDirectChildren`）。

- **分步、进度、绿态、持久化**：同第 7 节思路，标志位为 **`orderIntervalStdDevComputeCompleted`**，成功落盘后刷新仍绿底白字。  
- **与一次性全量 `applyOrderIntervalStdDevToDimension` 的区别**：该函数是**每级独立从底表**筛行算标准差、不分轮；**界面上**以分轮+克重树序+品名/规格**回退规则**为主路径，旧函数可保留作对照或他用。

### 8.2 「计算订货间隔平均值」与「生成周期性标签」

- **订货间隔平均值**：`applyOrderIntervalMeanToDimension`，各级别按该节点对应底表行子集**独立**求间隔的**均值**（与标准差**不同**的汇总策略）。  
- **周期性标签**：`applyPeriodicityToDimension`，`suggestPeriodicityLabel` 等由均值/标准差等推导文案。  
- 二者通常**单步**写回并保存，不以「多轮 rAF+完成态标志」为必需（以源码为准）。

## 9. 持久化

**模块**：`src/utils/salesForecastStorage.ts`

- **主 key**：`qifeng_sales_forecast_v1`，载荷版本 `v: 1`。  
- **内容**（同一次 `save` 中一致写入）：`preview`、**`analysisBase`（可空）**、**`orderSegments`（可空）**、**`customerInboundDimension`（可空）**、**`orderCountComputeCompleted`**、**`orderIntervalStdDevComputeCompleted`**。  
- **`SaveSalesForecastPersistedOptions`**：可单独传入上述两个**布尔**完成态；**未传入**的项从**当前已存**记录**合并**（一次 `load`），避免只更新 `customerInboundDimension` 时误把另一标志**清零**（实现见 `saveSalesForecastPersisted`）。  
- **视图 Tab key**：`qifeng_sales_forecast_view_tab_v1`，与主 payload 分离，只存 `user` / `debug`。  
- **老数据迁移**：如存在旧版平面 `skus` 等结构，在加载路径上通过 `migrateLegacySkusToModels` 等迁到五级树；无 `order*Completed` 字段时按**未**完成解析。

### 9.1 会重置两完成态的场景（典型）

- 新选并保存 CSV、拆解新底表、**重新「启动进货周期性分析」** 等，对持久化**显式**写入两标志 **`false`**（与内存态一致）。

---

## 10. 路由与切页

- **入口**：`App.tsx` 中 `SalesForecast` 在 `page === "salesForecast"` 时置 `active`；`active` 为 true 时**进入页面**会再次从 `loadSalesForecastPersisted` 同步，避免在应用内切页后**仅内存态**与**持久化**不一致。  
- **底表被清空/替换**时，联动清空数量分类、客户进货周期性分析数据、完成态等（见 `SalesForecast` 中 `useEffect` 对 `analysisBase`）。
